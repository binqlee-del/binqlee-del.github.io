一、课程核心结论

这节课解决的不是“怎样让 AI 写出一个页面”，而是页面生成之后如何真正运行、分享、交付和扩展。

完整路径是：

  1. 先判断项目目标；
  2. 再判断项目类型；
  3. 识别运行依赖；
  4. 选择启动、分享和部署方式；
  5. 需要模型、生图、数据库等能力时，再接入 API 或后端；
  6. 最后用 Skill 和合适的 AI 工具固化工作流。

讲师反复强调的原则是：项目目标不同，技术栈、提示词和交付方式也不同。不要先生成代码，再考虑怎么落地。

“不同的需求，你用的技术栈、你的提示词写的都不一样。”（约 04:00）

二、开始 AI Coding 前，先完成三个判断

1. 判断目标

课程把项目目标分成四档：

目标合理做法不必要的投入
快速验证想法本地可运行即可不必按生产标准搭建
可演示 Demo保证演示环境稳定不必过早建设完整后端
分享给别人使用静态化、局域网访问或部署不能只给 localhost 地址
交付研发或正式上线前置对齐技术栈、组件库、样式方案和接口不要生成一套与团队完全不兼容的代码

目标不清楚常见的后果：

  • 明明只是验证想法，却花大量时间处理工程问题；
  • 明明要交给研发，却生成了无法接入现有项目的代码；
  • 明明要分享给别人，却只在自己电脑的 localhost 上运行。

2. 判断类型

项目主要分为：

  • 纯静态前端：HTML、CSS、JavaScript；
  • 工程化前端：Vue、React 等；
  • 前后端分离项目；
  • 前后端不分离项目。

如果涉及数据库、账号、持久化数据或需要保护的密钥，通常已经不是纯前端项目，需要后端参与。

3. 判断依赖

“在某个平台里能预览”不等于“下载到本地还能运行”。平台可能暗中提供了：

  • 预览环境；
  • 大模型 API；
  • 生图 API；
  • 身份认证；
  • 数据存储；
  • 特定框架或组件库。

项目离开平台后，这些能力必须重新找到来源。课程举例说明：AI Studio 中可直接使用的模型或生图能力，下载到本地后可能只剩页面，相关功能会失效。

三、与研发交付时要前置对齐什么

如果项目最终要交给研发继续开发，至少要确认：

  • 框架：Vue、React，还是纯静态页面；
  • 组件库：应尽量与团队现有组件库一致；
  • 样式方案：CSS、Less、Sass、Tailwind CSS 等；
  • 项目结构和构建方式；
  • 是否需要后端、数据库或外部 API；
  • 团队已有的工程规范和部署环境。

课程中的典型反例：研发项目使用 Vue，而设计侧生成的是 React。页面即使视觉还原很好，迁移成本仍然可能很高。

提示词也应包含这些预置条件。例如，不只说“生成一个页面”，而是明确框架、样式方案、组件库、目标运行环境和需要接入的能力。

四、三类项目如何启动和交付

1. 纯静态页面

典型文件包括：

  • index.html
  • CSS 文件
  • JavaScript 文件

可以直接双击 HTML 打开，也可以在 VS Code 中安装 Live Server，用本地静态服务运行。后者更接近真实网页环境。

2. Vue / React 项目

这类项目需要安装 Node.js，并在项目目录运行开发命令：

npm run dev

验证 Node.js 和 npm 是否安装成功：

node -v
npm -v

开发地址通常表现为 localhost:端口号。端口号由当前项目和启动工具决定，不应写死理解为某一个固定值。

3. 打包交付

浏览器不能直接理解工程源码的全部形式。Vue、React 项目需要构建为浏览器可使用的 HTML、CSS、JavaScript 等静态资源：

npm run build

构建结果通常位于 dist 等目录。若接收方只需要部署或预览，应优先交付构建产物；若研发要继续开发，则应交付源码、依赖说明和启动方式。

五、把项目分享给别人：从临时到长期

方案 A：截图或录屏

适合只看效果，不需要操作。

方案 B：发送静态文件或构建产物

适合纯静态页面，或已经完成构建的工程化项目。注意：如果页面依赖 API、后端或跨域资源，只发送文件可能仍不能运行。

方案 C：局域网访问

localhost 和 127.0.0.1 都指向访问者自己的电脑，别人无法通过你的 localhost 地址访问你的项目。

“localhost 还有 127.0.0.1，指的是你自己的电脑。”（约 10:27）

在 macOS 上，课程演示了以下命令获取局域网地址：

ifconfig
ipconfig getifaddr en0

常见内网地址形态包括 192.168.x.x、10.x.x.x、172.16-31.x.x。启动服务后，用“局域网 IP + 端口”替换 localhost + 端口，同一网络中的其他设备即可尝试访问。

工程化项目还必须让开发服务器监听局域网接口；若只监听本机，即使替换 IP 也无法访问。课程演示中直接让 AI 修改项目配置，使 Vue 项目支持局域网访问。

局域网方案的限制非常明确：服务运行在个人电脑上，电脑关机、休眠或服务停止，访问就会中断。

“只要我的电脑一关，就没了。”（约 24:52）

方案 D：长期部署

长期访问有两类选择：

  • 自有服务器：可控，但需要成本和运维能力；
  • GitHub Pages、Vercel 等托管平台：适合静态站点和前端项目。

课程演示的 GitHub Pages 流程：

  1. 创建 GitHub 账号；
  2. 创建仓库；
  3. 在 Pages 中选择 GitHub Actions 作为部署来源；
  4. 让 AI 生成网页、提交代码和部署工作流；
  5. 通过公开地址访问。

重要安全边界：公司内部代码、数据、接口、密钥和其他敏感信息不能直接上传到公开 GitHub 仓库。课程明确提醒，应优先使用组织内部的资产平台或合规部署环境。

六、API：页面获得外部能力的方式

网页本身不会自动拥有大模型、生图或数据库能力。要完成这些任务，需要接入对应 API，或者建设后端服务。

课程给出的能力来源包括：

  1. 内部资产平台提供的 API；
  2. 官方模型或第三方服务 API；
  3. 本地代理或工具封装出的接口。

选择 API 时应关注：

  • 能力是否满足需求；
  • 接口是否稳定；
  • 计费和调用限额；
  • 数据是否会流向外部；
  • 密钥应该放在后端还是前端；
  • 是否符合公司安全规范和服务条款。

课程对低价第三方中转 API 的提醒很重要：服务商可能用低成本模型冒充高价模型，仅靠系统提示词声称自己是某个模型。不能只看返回文本里的“模型自报身份”，应通过可信来源、实际效果、计费和可验证的模型信息判断。

课程还演示了把已有 AI 工具能力封装成本地 API 的思路。这类方案虽然成本低，但实际采用前必须核对授权范围、账号安全、密钥管理和服务条款，不应默认等同于可用于生产环境的官方 API。

七、Skill 的核心设计原则

课程认为一个 Skill 最需要说明三件事：

  1. 在什么条件下触发；
  2. 能否被正确触发；
  3. 触发后具体做什么。

重点不在于堆积大量说明，而在于让 Agent 能稳定识别使用场景，并执行清晰、有限的任务。

课程引用了一组“Skill 数量与识别准确率”的测试数据：20 个以内约 90%、30 个以上约 70%、200 个约 20%。这组数字是讲师在课上转述的经验数据，视频中未给出原始研究来源，适合作为方向性提醒，不宜当作严格基准。

真正值得保留的结论是：

“Skill 这东西不是越多越好。”（约 45:37）
“如果你不知道这个 Skill 要不要用，那就不要它了；只有当你明确需要这个 Skill 的时候，才需要把它加进来。”（约 45:57）

这对应奥卡姆剃刀原则：如无必要，勿增实体。

八、工具选择：看场景，不看单一排名

课程画面给出的场景化选择如下：

工具更适合的场景课程备注
Comate内部环境、中文场景对组织环境更友好
Codex工程化开发、代码修改更适合技术同学或复杂项目
Cursor通用代码编辑与 AI 辅助取决于个人习惯

讲师口头推荐具有个人和时间背景，不应当作永久排名。更稳妥的选择标准是：能否使用现有账号和网络、是否支持团队工程规范、是否能访问需要的代码和工具、数据是否合规、协作成本是否可控。

十一、学习时最值得记住的五句话

  1. 先明确项目目标，再让 AI 生成代码。
  2. 交付研发时，技术栈一致比“单次生成效果惊艳”更重要。
  3. localhost 只属于自己的电脑，分享必须解决访问范围和部署问题。
  4. 平台内可用的能力，离开平台后可能全部变成外部依赖。
  5. Skill 和工具都不是越多越好，应围绕真实任务精简配置。
