关键前提：把 Agent 当作原型过程中的协作者

许多让人喜爱的交互和细节，都来自不断迭代、反复试错和仔细调整。原型设计可以帮助你快速尝试许多不同的想法。视频介绍的做法，是把 Xcode 中的 Coding Agents 和 Previews 结合起来，用它们制作关键页面与交互的原型。

视频强调

不要把批判性思考交给这些工具。你的任务仍然是运用自己的判断，为使用 App 的人打造尽可能好的体验。Coding Agent 应当被看作原型设计中的协作者，最终决定权始终在你手中。

因为 Agent 生成的是真实原生代码，所以这些原型代码可以继续用于后续开发。但工具的作用是帮助你发现更好的体验，而不是替你决定什么才是最好的体验。

开始之前：视频依赖的两个 Xcode 能力

Coding Agents

点击新建对话，然后描述希望实现的代码修改或功能，Agent 会完成相应工作。它让你可以仅通过描述想法，把想法变成代码。

Xcode Previews

无需每次修改后都重新构建和运行 App，就可以查看并操作界面。Swift 文件中需要先包含相应的 Preview。

两者结合：Agent 负责生成真实原生代码，Preview 让你立刻看到和操作结果；你负责判断哪些结果值得继续发展。

第一部分

探索 UI 的不同可能性

案例要做的是一个管理周期性书友会的 App。最初的提示词只有一句：“为一个定期聚会的书友会创建管理界面。”它可以很快生成结果，但由于提示词过于模糊，也带来了几个问题：

  1. 布局是任意选择的：这种组织方式可能可用，但它只是许多可能方式中的一种。
  2. Agent 猜测了功能：提示词没有清楚定义功能，因此生成了可能并不需要的投票和照片墙。
  3. 容易被有缺陷的起点限制：为了加入真正需要的功能，只能继续修改原有导航，最终可能出现功能膨胀，让界面显得笨重而不够优雅。
对于你的 App，以及它要解决的问题，你通常比 Agent 更清楚应该从哪里开始。

你很可能会喜欢不同方案里的不同元素。这时可以在后续提示词中说明哪些方案值得继续，以及希望把哪些具体元素重新组合。视频中的做法，是列出各个方案里喜欢的功能，并再次要求每次迭代都有自己的 Swift Preview 和独特名称。

第二部分

引入真实内容，让 App 感觉有人用过

在开发早期，最好让不同的人试用 App 并提供反馈。但有些 App 还没有准备好让人们放入自己的内容。这时，Agent 可以暂时扮演使用 App 的人，把只有少量占位内容的空白模板，变成内容丰富、更加接近真实使用的界面。

视频给出了几项需要主动检查的内容：

  • 不要只创建一个 Preview，而是创建许多 Preview。
  • 自己先想清楚边界情况，例如还没有安排聚会时，详情区域应该是什么样子。
  • 明确指出要改变哪些界面状态，避免 Agent 遗漏。
  • 提供足够上下文，让样例内容对目标用户来说可信；书友会里的讨论应该围绕书籍。
  • 注意会无限增长的内容，例如成员数量、消息长度、往期书籍数量和长文本输入。
  • 让样例内容放在独立、可读且可复用的文件中，方便继续修改。

另一个问题是：参与者太多时，排行榜会变得很长，人们需要滚动很久才能看到讨论。视频的处理方式是始终显示当前用户的相对排名，并通过展开控件查看完整列表。这些例子说明，真实内容可以帮助你理解人们实际使用 App 时可能遇到的体验。

视频提示词的中文翻译：生成不同内容状态 
请创建一个新的 Swift 文件，展示这个设计在以下情况下的不同版本：长文本和短文本输入、大量和少量书友会成员，以及其他我可能没有想到的界面状态。

确保这些版本共享可复用的模型，让我可以阅读和修改它们，也让未来的其他视图可以继续使用这些模型。

请为每个版本创建独立的 Swift Preview，并分别给出描述性的名称。

注意：没有什么能替代现实中的使用，以及真正试用 App 的人所提供的反馈。生成内容的作用，是帮助你更早进入原型设计的反馈阶段。

第三部分

调整 App 中重要的动画与交互

静态界面确定后，视频进入交互、动画和转场。常见动画主要包括：

Ease：平滑加速、减速或两者兼有，主要调持续时间与曲线。

Spring：模拟弹簧力，核心参数是 stiffness、damping 和 mass。

物理反馈：拖拽还涉及摩擦与惯性；设备运动和触觉可以传达重量、距离和模式变化。

可以直接在代码中修改常量并用 Preview 观察结果，但相关常量有时分散在代码的不同位置，来回切换会很麻烦。视频建议创建一个专用界面，用来调整与当前交互有关的参数，并让 Agent 帮助生成这个调参面板。

把动画拆成不同阶段

Phase 1书籍网格进入详情页视图转场到详情页面
→
Phase 2详情内容依次出现后续各行内容按交错时间进入

把动画拆成阶段，可以让调参面板更容易理解，也让你和 Agent 能用共同的词汇讨论动画的具体部分。每个阶段还可以单独查看。调参面板不只适用于动画参数，也可以切换 App 状态、颜色、字体样式或视觉偏移。

视频提示词的中文翻译：创建动画调参面板 
请创建一个专用视图，用来管理书籍网格与详情页之间的动画。

动画包含两个阶段。Phase 1 是书籍网格与详情页之间的转场。Phase 2 是详情页出现之后触发的一组动画。

请为动画曲线、参数、时间和延迟提供多个选项，并允许在这个专用视图中单独播放每个阶段。

请在独立文件中创建一个视图：当窗口宽度大于 compact（例如 iPad）时，让 App 界面和控制面板并排显示。请为它创建 Swift Preview。

视频最后总结的做法

  1. 01
    提示词要具体

    先想清楚功能和高层次重点，并提供希望 App 传达的风格与感觉。

  2. 02
    要求多个方案

    早期最适合探索多个不同方向；为每个方向创建独立、具名的 Swift Preview。

  3. 03
    选择元素并重新组合

    指出哪些方案有潜力，以及希望把哪些具体元素组合到一起。不断扩大探索、重新组合、继续重复。

  4. 04
    引入真实内容和边界情况

    检查空状态、长文本和会不断增长的内容；让样例模型可读、可修改并可复用。

  5. 05
    为需要调整的内容创建控制面板

    说明要调整的参数，把复杂动画拆成阶段，并在宽窗口中把控制面板和界面并排显示。

  6. 06
    仍然需要真实反馈与人的判断

    没有什么能替代实际使用 App 的人所提供的反馈。最终决定权始终在你手中。

视频中的概括

Go wide. Remix. Repeat.

广泛探索、重新组合、不断重复。整个过程最关键的一块，仍然是你的判断。
